What it does

The whole profile-HMM pipeline, orchestrated behind one UI

HMM Discovery drives the standard bioinformatics CLIs — HMMER, MAFFT, trimAl, Prodigal, IQ-TREE, MEME — from a single guided app, so you get the sensitivity of a profile HMM without hand-running every command.

🧬

Seed to profile HMM, guided

Load a curated protein FASTA, GenBank, nucleotide FASTA, or a whole folder. The app aligns your seeds with MAFFT, trims poorly conserved columns with trimAl, and builds an HMMER3 profile with hmmbuild — no flags to remember.

✅

Seed self-recovery before you scale

Step 5 searches your seeds back against the new HMM so you can confirm strong self-recovery before spending CPU on large databases. Weak recovery usually means the seed set needs curation — and the app tells you so.

🗄️

Public database search, built in

Search INPHARED genomes and proteins, RefSeq viral genomes/proteins, Swiss-Prot, GPD, GVD-AVrC, and RefSeq bacterial proteins for host/background specificity — all with resumable, provenance-tracked downloads.

🔬

Six-frame ORF vs Prodigal

For nucleotide databases, choose exhaustive six-frame ORF scanning — every stop-to-stop peptide, forward and reverse — to catch short, overlapping, or annotation-missed genes, or fast Prodigal gene prediction as a conventional baseline.

📊

Confidence-tier hit classification

Every hit is scored on multiple lines of evidence — bit score, HMM coverage, reciprocal/self-search behavior, domain match, and QC flags — into four tiers: high-confidence, putative, divergent, and likely false positive.

🏷️

Pfam & VOGDB annotation

Discovered proteins are scanned against Pfam-A for broad conserved domains and against VOGDB release 230 VFAM HMMs (39,585 viral ortholog/family models) with hmmscan — same reproducible path across laptop, workstation, and container.

🧭

Synteny neighborhood recovery

The app recovers five upstream and five downstream genes around each placed hit and writes synteny tables, a placement report, and a GFF3 you can drop straight into clinker, pyGenomeViz, or Easyfig-style tools.

🌳

IQ-TREE phylogeny

Build a phylogeny of your hits with IQ-TREE — model selection and bootstrap support included — to see how discovered members relate to your seed family, right inside the analysis step.

🧩

Motifs & clustering

Run MEME/FIMO for motif discovery and scanning, and CD-HIT or MMseqs2 for sequence clustering, as downstream interpretation layers over your discovered set — when the tools are installed and you ask for them.

📦

Reproducible ZIP export

Step 9 turns a run into a package: hit tables, presence/absence matrix, taxonomy table, synteny GFF3, an HTML report, plus METHODS_TEXT.txt and reproducibility.json — the methods paragraph and provenance written for you.

📁

Native file & folder pickers

Every path field has a Browse-style chooser. On desktop installs, Choose File and Choose Folder open the real OS picker; an in-page navigator and manual path entry are always there as fallbacks. No copy-pasting long paths.

🧪

Synthetic demo, one click

A synthetic demo FASTA ships with the app so a brand-new lab member can create a project, align, build an HMM, confirm seed recovery, and export a ZIP — verifying the whole environment before real data enters the system.

⚙️

Same pipeline on the command line

Every step is also a standalone script in scripts/, driving the same pipeline/ code — interactive in a terminal, silent with --yes — for HPC clusters, headless servers, and copy-pasteable Methods sections.

🛡️

Resumable, disk-aware runs

Large searches cache compressed public files first, then translate and search locally, so a dropped connection doesn't destroy a long run. Completed databases are skipped on resume, and raw caches are cleaned between chunks to spare disk. Install with conda, one docker run, or the launcher script.

Why this exists

The story behind HMM Discovery

The predicament

Profile-HMM discovery is one of those techniques that is genuinely powerful and genuinely miserable to run. The idea is beautiful: instead of asking whether one sequence looks like another, you build a statistical model of a whole protein family and go looking for its distant relatives — the divergent members a plain BLAST search quietly walks past. For anyone who studies phage proteins, where the interesting genes are often short, fast-evolving, and unannotated, that sensitivity is exactly what you want.

The misery is in the plumbing. To actually do it, you chain together half a dozen command-line tools, each with its own conventions. You align seeds with MAFFT. You trim columns with trimAl. You build a profile with hmmbuild and search with hmmsearch. If your database is nucleotide, you translate it first — six frames if you are thorough, Prodigal if you are in a hurry. You parse tblout files. You write a script to classify hits. You run IQ-TREE for a tree, maybe MEME for motifs. And somewhere around the fourth tool, you lose track of exactly what you ran.

By the end you have your answer and no clean record of how you got there — a folder of intermediate files and a shell history you would not want a reviewer to read.

So I did what I have started to do whenever I hit a wall like this in the age of AI: I remembered that you can just build things.

The actual problem with the workflow

To be clear, none of these tools are bad. HMMER is a masterpiece. MAFFT, trimAl, Prodigal, IQ-TREE, MEME — these are the load-bearing software of modern sequence biology, built by people who deserve enormous credit. The problem is not any single tool. The problem is the seams between them.

Profile-HMM discovery is a specific workflow — align, trim, build, search, translate, classify, recover context, annotate, export — and no tool in the standard stack owns it end to end. You are the integration layer, holding in your head which E-value you used, which database version you pulled, whether you ran six-frame ORFs or Prodigal on that one genome, and where the intermediate files went. That load is invisible when everything works and catastrophic when a reviewer asks you to reproduce a run three months later.

And there is a second cost that bothers me more as a scientist: the barrier keeps good people out. A talented bench biologist with a curated seed alignment should not need shell scripting to search it against INPHARED. The technique is not the hard part. The plumbing is the hard part, and the plumbing is not the science.

What I wanted was almost embarrassingly simple to state. I wanted a colleague to open a browser, drop in their seed FASTA, click through the pipeline, and walk away with reproducible output — including the methods paragraph, already written, ready to paste into a manuscript.

What I built

HMM Discovery is a Shiny app written in Python. It walks you through nine steps: load your input, build and inspect the alignment, build the profile HMM, search databases, validate seed recovery, refine, review results, run downstream analysis, and export. You do not write Python, R, or shell commands to do any of it. Under the hood it is orchestrating the same trusted CLIs you would have run by hand — it just remembers the flags, the versions, and the order for you.

A few decisions I care about. Nucleotide searches offer exhaustive six-frame ORF scanning as a first-class discovery mode, not an afterthought — because the phage genes I care about are often the ones a conventional gene caller skips. Hits are sorted into four honest confidence tiers instead of a single yes/no, with the reasoning attached, because triage is the real job and I would rather the app show its work than pretend to certainty. Synteny recovery pulls the five genes on either side of each hit, because in phage biology genomic neighborhood is evidence.

And the export step is the part I am quietly proudest of. Every run can produce a METHODS_TEXT.txt, a reproducibility.json recording tool versions, database URLs, and access dates, a run summary, the hit tables, the synteny GFF3, and a single ZIP you can hand to a collaborator or attach to a paper. The reproducibility is not a feature I bolted on. It is the whole reason the thing exists.

On building tools as a scientist

There is a school of thought that treats tool-building as a distraction from real science. Get the data, analyze it, publish, move on. And there is truth in it — hours spent on infrastructure are hours not spent at the bench.

But I have come to believe, more firmly as my PhD has gone on, that building a tool forces you to understand a problem at a depth that merely using tools never does. Writing the confidence classifier meant deciding, explicitly, what distinguishes a real distant homolog from a low-complexity artifact — bit score, coverage, reciprocal behavior, bias. Wiring up six-frame translation meant confronting how many candidate ORFs you generate and why that sensitivity has to be paid for with review time. You cannot hand-wave any of that when it has to run.

Building the tool made me a more careful reader of my own results. That alone would have justified the time.

I am not claiming HMM Discovery is finished, or that it removes the need for judgment. It is a discovery and triage instrument, not an oracle — the biological claims still have to be earned from the hit tables, the coverage, the genomic context, and appropriate follow-up. But it runs, it is reproducible, and it has already saved the people around me more time than it took to build. That feels like the right bar for whether something was worth doing.

A note on how it was built

HMM Discovery was built collaboratively with Claude (Anthropic's AI). The scientific decisions are mine: which databases matter, why six-frame ORFs deserve to be a first-class mode, what belongs in a confidence tier, what a reproducible export has to contain, and the frustration that started the whole thing. Claude was the implementation partner — writing and debugging the Shiny UI, the job runners, the search and classification code, the synteny and export logic. I am good at building tools with AI; I am not a career software engineer, and it is worth being straightforward about that. Working this way let me turn a specific, well-understood annoyance into a working, reproducible tool far faster than I could have alone.

Getting started

From seed sequences to a reproducible export

The app is guided end to end. Run the synthetic demo first to confirm your environment, then point it at your own protein family.

01

Launch and open a project

Install with conda or Docker, run the launcher, and open http://127.0.0.1:8081. Create a new project directory — everything for this run lives there.

02

Load your seeds

Point the input step at a curated protein FASTA, GenBank, nucleotide FASTA, or a folder. Use the native file picker — no path typing required. Try the bundled demo FASTA on your first run.

03

Align & build the HMM

Run MAFFT alignment and trimAl trimming, then build the HMMER3 profile. Check seed self-recovery in the validation step before you commit CPU to large searches.

04

Choose databases and search

Search INPHARED proteins first for a fast check, then add INPHARED and RefSeq viral genomes. For nucleotide targets, pick exhaustive six-frame ORFs or Prodigal.

05

Classify, annotate, analyze

Review hits by confidence tier, annotate against Pfam and VOGDB, recover synteny neighborhoods, and — when you want them — run an IQ-TREE phylogeny, motifs, or clustering.

06

Export the whole package

Step 9 writes hit tables, figures, an HTML report, METHODS_TEXT.txt, reproducibility.json, a run summary, and a single ZIP — provenance and methods text included.
