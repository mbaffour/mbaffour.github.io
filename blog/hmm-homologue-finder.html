What it does

The whole discovery loop, packaged into one command

Every step of a remote-homologue search — build, search, validate, iterate, characterise, package — automated, reproducible, and backed by tests so a code change can't quietly change the biology.

⌘

One command, seed FASTA in, package out

Give it a seed FASTA and it does the rest: builds the HMM, picks and searches the databases, validates hits, iterates, and assembles a labelled output folder. On first run it even installs the bioinformatics tools it needs via conda.

🧬

Profile HMMs reach the twilight zone

Position-specific scoring weights a conserved catalytic residue heavily and a floppy loop loosely, detecting homology down to ~15–25% identity — the remote range where pairwise BLAST quietly fails.

🔃

Six-frame search finds the unfindable

Genome databases are translated in all six frames, so homologues encoded antisense or out-of-frame to predicted genes — the ones standard annotation misses entirely — become searchable. Some proteins are found only this way.

✅

ORF validation proves they're real genes

Each hit is reconstructed frame-correctly from genomic coordinates, required to carry no internal stop codons, and checked to sit in a real coding locus. An HMM score alone is never enough — the evidence lands in the table.

🔁

Iterate to convergence

Deduplicate hits, re-seed the model, search again — and stop early, automatically, when the hit set and the model stabilise. The pipeline records the round-by-round counts and the reason it stopped.

🎯

Controls + ROC quantify the false-positive rate

Composition-matched shuffled seeds and unrelated proteomes calibrate the bit-score threshold, giving sensitivity, specificity, FPR and an ROC AUC — so "it's just composition bias" can be answered with a number, not a shrug.

🔬

Golden-file regression tests

A stored baseline snapshots the biological invariants that must stay stable for a fixed seed and pinned databases — the representative accessions, confidence tiers, bit scores, ORF invariants, and tree topology — so a refactor can't silently move the answer.

🔒

Locked, pinned environment

An environment.lock.yml pins every tool — HMMER 3.4, MAFFT, trimAl, Prodigal, CD-HIT, IQ-TREE, MEME/FIMO, clinker — and each run also records the live versions it actually used in its manifest.

📊

Evidence tables, not just sequences

A 37-column hits.tsv gives organism, genomic coordinates, ORF-validation metrics, HMM statistics, and both nucleotide and amino-acid sequence per hit — plus a deduplicated table with a "found in N databases" provenance column.

🌳

Alignment, tree, and synteny — publication-ready

A high-quality MAFFT L-INS-i alignment, an IQ-TREE ML tree with the seeds placed and marked, and per-cluster gene-neighbourhood figures — all written as editable SVG and PDF alongside a 300-dpi PNG.

🧩

Overprinting, demonstrated — not just suspected

The opt-in interrupted-homolog search reads through premature stops, then tests whether each stop is synonymous in an open antisense frame — the sequence signature of overprinting — and reports a strong / partial / none verdict per candidate.

🧭

Single-genome scan mode

Ask "does this genome carry my gene, and where?" Scan one genome — a local FASTA or an NCBI accession, fetched for you — and get present / absent, coordinates, ORF validation, the flanking genes, and a genome-map figure with your gene in bold gold.

📦

A self-contained, provenance-rich package

Every run assembles a labelled PACKAGE/ folder with a run_manifest.json, a written METHODS.md, GFF3 browser tracks, named GenBank neighbourhoods, the profile .hmm, and an HTML results summary — everything needed to reproduce and to publish.

💻

macOS, Linux, and Windows via WSL2

The engine is bundled in the repo, so it runs anywhere you clone it. Double-click on a Mac, bash run.sh on Linux, or run inside WSL2 on Windows — the bioinformatics tools ship only for macOS and Linux, so Windows goes through WSL2.

📡

Streamed, cached databases

Databases are streamed and six-frame-translated on the fly, then cached so later runs reuse them. Preload them ahead of time when you have spare bandwidth, and later searches spend their time searching, not downloading.

Why this exists

The story behind HMM Homologue Finder

In this post

The hand-stitched loop The actual problem What I built On building tools Built with Claude

Hunting for remote homologues of a protein family is one of those tasks that looks simple written down and turns into a day of babysitting a terminal. You build an alignment of your seeds. You make an HMM from it. You search a database. You pull the hits. You reconstruct and validate their ORFs. You feed the good ones back in as new seeds, and you search again. Then you repeat that loop until nothing new comes back.

Each of those steps is a different tool with its own flags, its own file formats, and its own quiet failure modes. And by the third iteration, the honest truth is that you have lost track of exactly what you ran — which E-value cutoff on iteration two, which databases in iteration three, whether you remembered to re-translate the genome set in all six frames or just searched the annotated proteins. The pipeline lives in your shell history and your short-term memory, and both are lossy.

By the third round, the pipeline lived in my shell history and my short-term memory — and both are lossy.

This matters more in phage biology than almost anywhere, because so much of the interesting stuff is hidden. Some of the proteins I care about are encoded antisense or out-of-frame to a predicted gene, so a search over annotated proteins never sees them. You have to translate the raw genome in all six frames to reach them — and once you are doing that by hand across a dozen databases and several iterations, the bookkeeping is the hard part, not the biology.

The actual problem

To be clear, none of the underlying tools are bad. HMMER, MAFFT, Prodigal, CD-HIT, IQ-TREE, clinker — these are excellent, battle-tested pieces of software maintained by people who deserve enormous credit. The problem is not any single tool. The problem is the seam between them.

A remote-homologue search is a specific, repeatable workflow: build a profile, calibrate a threshold against controls, search a set of databases in six frames, reconstruct and validate every hit as a real ORF, deduplicate, re-seed, and iterate to convergence. But stitching that together by hand means you are constantly leaving one program and entering another, re-importing files, and re-deriving decisions you already made yesterday. The cost is not really the time. It is the fragmentation. Your analysis exists simultaneously as a shell script, a pile of intermediate FASTAs, a mental model of the cutoffs, and a vague memory of the order you did things in.

And fragmentation is the enemy of reproducibility. When a collaborator asks "how did you find this one?" — or when you ask yourself six months later — the answer has to be reconstructable from artifacts, not from memory. If a hit is going into a figure or a manuscript claim, "I'm pretty sure I ran it like this" is not good enough.

What I built

HMM Homologue Finder packages that entire loop into a single command. You give it a seed FASTA. It builds the HMM, calibrates the score threshold against positive and negative controls, searches the database catalog in six frames, reconstructs and validates each hit's ORF, deduplicates across databases and iterations, re-seeds, and repeats — stopping automatically when the hit set and the model stabilise. Then it characterises what it found: a high-quality alignment, an ML tree with the seeds placed and marked, gene-neighbourhood synteny figures, motifs, GFF3 tracks, named GenBank files, and a self-contained results package with a written methods file and a full manifest.

Two design decisions mattered most to me. The first is that it produces the same answer every time for a fixed seed and pinned databases. The environment is locked, the stochastic steps are seeded, and there is a golden-file regression test that snapshots the biological invariants — the representative accessions, the confidence tiers, the bit scores, the ORF invariants, the tree topology — so that a refactor cannot silently move the biology. If a code change would alter which homologues you find, a test fails and tells you. That is the whole point.

The second is that it treats a hit as a candidate that has to earn its place. Every hit is reconstructed frame-correctly, required to have no internal stop codons, and checked to sit in a genuine coding locus. Controls and an ROC curve quantify the false-positive rate, so "it's just composition bias" can be met with a number. A validated hit is still only a candidate — function needs experimental work — but at least it is one you can defend.

On building tools as a scientist

There is a version of scientific training that treats tool-building as a distraction from real science: get the data, analyse it, write it up, move on. And there is real wisdom there — time spent on infrastructure is time not spent at the bench.

But there is another version I have come to believe more fully as my PhD has gone on: the act of building the tool forces you to understand the problem at a depth that using tools never does. Writing the ORF-validation step meant confronting exactly what it means for a six-frame hit to be a "real" gene — where the frame is, where the stops are, whether the locus is genuinely coding. Writing the overprinting test meant being precise about a signature I had been waving my hands at: a premature stop that is synonymous in an open overlapping antisense frame. You cannot fudge that in code. The compiler, and the test suite, make you say exactly what you mean.

Building the tool forces you to understand the problem at a depth that using tools never does.

I am not claiming this is finished. It finds homologues by sequence and HMM, not structure; from assembled databases, not raw reads; and read-based expression evidence is planned, not wired in. It has gaps I know about and surely gaps I do not. But it is real, it runs, it is reproducible, and it has already saved me more time than it took to build. That feels like the right bar for whether something was worth doing. You can just build things.

A note on how it was built

This was built collaboratively with Claude (Anthropic's AI). The science, the workflow design, the frustration that motivated it, and the decisions about what to build and how to validate it are mine. Claude was the implementation partner: writing and debugging the pipeline glue, the six-frame extraction and ORF-reconstruction logic, the control-calibration and regression-test harness, and the figure generation. Working this way let me turn a specific, real research frustration into a reproducible tool far faster than I could have alone, and I think it is worth being transparent about that.

Getting started

Clone it, set it up, point it at your seeds

macOS, Linux, or Windows via WSL2. The first setup creates a locked conda environment; after that, a run is one command. Full instructions live in docs/INSTALL.md and docs/USAGE.md.

01

Clone the repository

Run git clone and cd into it. The search engine is bundled in engine/, so the repo is self-contained — there's nothing else to download to make it run.

02

Set up once

Run bash setup.sh. It creates the hmm-discovery conda environment and installs every tool — HMMER, MAFFT, Prodigal, IQ-TREE and the rest — at pinned versions. One-time only.

03

Smoke-test the install

Run the example seeds with --smoke to confirm everything is wired up before you commit to a real, database-downloading run. Fast, offline, and a good first-machine check.

04

Preload the databases

Optionally run preload_databases.py when you have spare bandwidth. Databases stream, get six-frame-translated, and cache — so later searches spend their time searching, not downloading.

05

Run discovery

Run bash run.sh and hand it your seed FASTA. It builds the HMM, searches the databases you pick, validates ORFs, and iterates to convergence — no babysitting required.

06

Open the package

Everything lands in a labelled PACKAGE/ folder: the evidence tables, alignment, tree, synteny figures, GenBank files, the profile .hmm, a written METHODS.md, and an HTML results summary.
